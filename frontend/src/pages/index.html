§ 01 — Overview

Feature Summary

This feature replaces the existing single-path sign-up and sign-in with a designation-aware flow. Operators share an email address and are created in bulk; all other roles follow a unique-email path and may be assigned to multiple plants.

ℹ
Current state: The existing Auth.tsx page collects only email + password with no designation step. The /admin/users/create endpoint creates all users with a default Operator role and an empty plant_assignments array. This spec defines all changes required.

Scope of Changes

  • FE-01New multi-step sign-up flow in Auth.tsx with designation-first branching
  • FE-02Bulk Operator form: dynamic N-user input set generation from a count input
  • FE-03Operator sign-in: email → username picker before proceeding
  • FE-04Admin UsersPanel: single-plant enforcement for Operators in PlantAssignmentEditor
  • FE-05OperatorSwitcher RBAC gate extended to cover all 6 designations
  • BE-01POST /admin/users/create extended to accept plant_assignments and operator_count
  • BE-02New POST /auth/signup route for self-service sign-up with designation branching
  • BE-03Audit tables: signup_audit, extended login_attempts, plant_assignment_log
  • DB-01Schema: user_profiles.plant_assignments enforces max-1 constraint for Operators via trigger
§ 02 — Designations

Supported Roles & Constraints

Six designations are supported. They differ in email uniqueness rules, plant assignment cardinality, and whether operator switching is permitted.

Designation Email Plant assignment Operator switch EnumList (multi-plant UI) Sign-in path
Operator Shared Single plant only ✓ Allowed ✗ Disabled Email → username picker
Manager Unique Multi-plant ✗ Blocked ✓ Enabled Email + username
Admin Unique Multi-plant ✗ Blocked ✓ Enabled Email + username
Maintenance Unique Multi-plant ✗ Blocked ✓ Enabled Email + username
Quality Assurance Unique Multi-plant ✗ Blocked ✓ Enabled Email + username
Data Analyst Unique Multi-plant ✗ Blocked ✓ Enabled Email + username
⚠
Shared email constraint: Multiple Operators share one Supabase auth.users row (same email). Each Operator has their own user_profiles row with a unique username. The sign-in session belongs to the email; the active profile is selected post-login via the username picker. This is a significant departure from Supabase's default 1-email-1-user model — see Backend Changes §06 for the implementation approach.
§ 03 — Sign-Up Flow

Multi-Step Registration

Sign-up is restructured into a wizard. Designation is always collected first; subsequent steps are determined by whether the designation is Operator or not.

Step 1 — Designation Selection (all users)

1
Ask designation
Present a dropdown or segmented selector with all 6 designations. No other fields are shown at this point. User cannot proceed until a designation is chosen.

Component: replace the current sign-up tab in Auth.tsx with a stepper. Step indicator shows 1 of 2 (or 1 of 3 for Operators).

Step 2 — Branching by Designation

Operator Branch
  • Ask: "How many Operators will use this email?"
  • User enters a count N (integer, min 1, max 20).
  • System renders N input sets, each containing:
    Username · First Name · Middle Name (opt) · Last Name · Suffix (opt) · Password
  • All N usernames are tied to a single shared email field shown above the sets.
  • A single Plant selector (single-select, not multi) is shown once below all sets — all Operators in this batch are assigned the same plant.
Non-Operator Branch
  • Standard single-user form:
    Email · Password · Username · First Name · Middle Name (opt) · Last Name · Suffix (opt)
  • Plant assignment via multi-select EnumList — user may pick one or more plants. A "Skip for now" option is available (Admin can assign later).
  • No operator count input is shown.
  • No shared-email mechanics.

Step 3 (Operators only) — Review & Submit

Show a read-only summary of all N operator records before submission. User can go back to edit. On confirm, all N profiles are created in a single backend call.

✓
Post-submission: All newly created accounts (both flows) land in status = 'Pending'. An Admin must approve before the account becomes active. The pending-approval screen at /pending-approval is shown immediately after sign-up for non-Admin-created accounts.

Validation Rules

Email
Valid RFC-5321 email · max 255 chars · unique per non-Operator registration
Password
Min 8 chars · max 72 chars (bcrypt limit)
Username
Alphanumeric + underscore · min 3 chars · globally unique across user_profiles
Operator count
Integer 1–20. Submitting count 1 still goes through the Operator flow (single shared-email user).
Plant (Operator)
Exactly one plant required. Cannot be empty before submission.
Plants (non-Operator)
Zero or more (optional at sign-up). Admin can assign after approval.
§ 04 — Sign-In Flow

Authentication Paths

Sign-in differs depending on whether the email belongs to a shared Operator group or a unique non-Operator account. The system determines this automatically from the email entered.

Path A — Operator (shared email)

1
Enter email + password
Standard Supabase signInWithPassword. Authenticates the shared auth user.
2
Username picker
After successful auth, the system queries user_profiles for all profiles sharing the same auth.users UID whose plant_assignments overlaps the session's plant context. A list of Active Operators is shown. User selects their username.

This step reuses the OperatorSwitcher paradigm but surfaces it at login rather than post-login.
3
Session established with selected profile
The selected user_profile.id is stored in appStore.activeOperatorId. All subsequent data writes use the active operator's profile ID, not the auth UID.
⚠
Cross-plant restriction: An Operator can only select usernames whose plant_assignments matches the plant currently in scope. The PlantID match enforced in OperatorSwitcher.handleSelect() applies here identically. Attempting to select a username from a different plant returns an error.

Path B — Non-Operator (unique email)

1
Enter email + password
Standard Supabase signInWithPassword. No additional username picker step.
2
Session established directly
The user_profiles row for the authenticated UID is loaded. activeOperatorId is set to null (own profile is always active). The operator-switch section in OperatorSwitcher is hidden.

How the system detects Operator vs. non-Operator on sign-in

After step 1 succeeds in either path, the frontend queries:

-- Called immediately after signInWithPassword resolves
SELECT designation, plant_assignments
FROM user_profiles
WHERE id = '<auth.uid()>'

If designation = 'Operator', the username picker is shown. Otherwise, the user proceeds directly to the dashboard.

§ 05 — Admin Console

Admin Console Enhancements

The existing UsersPanel and PlantAssignmentEditor components require changes to enforce designation-specific plant constraints and expose multi-plant assignment for non-operators.

CreateUserDialog — Extended Fields

The existing dialog in UsersPanel.tsx needs two additions:

  • FE-04aOperator count field — visible only when designation = 'Operator' is selected. Integer input, default 1, range 1–20. Renders N username/name rows dynamically below.
  • FE-04bPlant assignment field — always shown. For Operators: single-select (radio list or searchable single-select). For all other roles: multi-select checkboxes (the existing PlantAssignmentEditor behavior). This is now exposed inline in the create dialog, not deferred to a post-creation edit.
  • FE-04cMandatory for Operators: submission blocked if no plant is selected when designation = 'Operator'.

User Detail View — Plant Display

The user card row in UsersPanel currently shows a count string (e.g. "2 plants assigned"). Replace with:

  • FE-05aFor Operators: show the single assigned plant name as a badge. If none, show No plant — action required.
  • FE-05bFor non-Operators: show up to 3 plant-name badges inline, then +N more if count exceeds 3.
  • FE-05cThe "Edit plants" button in PlantAssignmentEditor should pass a maxSelections={1} prop when the user's designation is Operator, which disables all checkboxes when one is already checked.

§ 06 — RBAC Enforcement

Role-Based Access Control

RBAC gates must be updated to cover all 6 designations. The existing logic in OperatorSwitcher.tsx only checks for 'Operator'; the new model must explicitly block non-operator roles from the switch mechanism and from receiving EnumList multi-select.

Operator Switch Visibility

Designation Can see "Switch operator" Condition
Operator ✓ Yes designation === 'Operator' AND plant_assignments overlaps target
Manager ✗ Hidden Always hidden
Admin ✗ Hidden Always hidden
Maintenance ✗ Hidden Always hidden
Quality Assurance ✗ Hidden Always hidden
Data Analyst ✗ Hidden Always hidden

Updated canSwitchOperator Constant

No logic change needed — the existing check already only allows 'Operator'. Make the intent explicit by updating the constant name and adding a comment:

// OperatorSwitcher.tsx — no functional change; rename for clarity
const OPERATOR_ONLY_DESIGNATIONS = ['Operator'] as const;

function canSwitchOperator(profile: Profile | null): boolean {
  if (!profile?.designation) return false;
  // Explicitly blocked: Manager, Admin, Maintenance, QA, Data Analyst
  return OPERATOR_ONLY_DESIGNATIONS.includes(
    profile.designation as typeof OPERATOR_ONLY_DESIGNATIONS[number]
  );
}

EnumList Multi-Plant Assignment Gate

// Helper used in both UsersPanel and PlantAssignmentEditor
const NON_OPERATOR_DESIGNATIONS = [
  'Manager', 'Admin', 'Maintenance', 'Quality Assurance', 'Data Analyst',
] as const;

export function isMultiPlantEligible(designation: string | null): boolean {
  if (!designation) return false;
  return (NON_OPERATOR_DESIGNATIONS as readonly string[]).includes(designation);
}

Backend RBAC Guard on Plant Assignment

The /admin/users/{id}/update-plants endpoint (new) must validate that Operator accounts receive at most 1 plant ID in the payload. Return HTTP 422 if more than one is provided.

# server.py — plant assignment validation
if profile.get("designation") == "Operator" and len(body.plant_ids) > 1:
    raise HTTPException(
        status_code=422,
        detail="Operators may only be assigned to a single plant.",
    )
§ 07 — Backend Changes

API & Server Updates

Two existing endpoints require extension and two new endpoints must be added to support the shared-email Operator model and the new sign-up flow.

Extended: POST /admin/users/create

POST /admin/users/create

Extend AdminCreateUserRequest to accept bulk Operator creation and plant assignments:

class AdminCreateUserRequest(BaseModel):
    email:        str
    password:     str              # Used for all Operator accounts in batch
    designation:  Optional[str]    = None
    plant_ids:    List[str]        = []   # 1 item max if designation=Operator

    # Single-user path (non-Operator or single Operator)
    username:     Optional[str]    = None
    first_name:   Optional[str]    = None
    last_name:    Optional[str]    = None
    middle_name:  Optional[str]    = None
    suffix:       Optional[str]    = None

    # Bulk Operator path — overrides single-user fields when present
    operators:    Optional[List[OperatorRecord]] = None

class OperatorRecord(BaseModel):
    username:    str
    first_name:  str
    last_name:   str
    middle_name: Optional[str] = None
    suffix:      Optional[str] = None

When operators is provided, the endpoint creates a single auth.users row (shared email) and then upserts N user_profiles rows, each with the same plant_assignments = [plant_ids[0]].

New: POST /auth/signup

POST /auth/signup

Self-service sign-up endpoint replacing direct Supabase signUp() calls from the frontend. This allows server-side designation branching and audit logging before the account is created.

class SignupRequest(BaseModel):
    email:       str
    password:    str
    designation: str             # Required — determines branching
    plant_ids:   List[str]       # 1 for Operator, 0+ for others

    # Non-Operator single user
    username:    Optional[str]   = None
    first_name:  Optional[str]   = None
    last_name:   Optional[str]   = None
    middle_name: Optional[str]   = None
    suffix:      Optional[str]   = None

    # Operator bulk path
    operators:   Optional[List[OperatorRecord]] = None

New: GET /auth/operator-usernames

GET /auth/operator-usernames?email={email}&plant_id={plantId}

Called after a successful Supabase sign-in when the system detects the account is Operator-designated. Returns all Active Operator profiles linked to the shared email whose plant assignments match the given plant ID.

# Response shape
{
  "operators": [
    {
      "id": "uuid",
      "username": "jdelacruz",
      "first_name": "Juan",
      "last_name": "Dela Cruz",
      "plant_assignments": ["plant-uuid"]
    }

}

Extended: POST /operator/switch-log

No schema change. Add login_type: "initial_selection" | "mid-shift_switch" field to distinguish first-login operator selection from a subsequent switch during a session.

ℹ
Shared-email implementation note: Supabase does not natively support N profile rows per auth user without custom logic. The workaround: the shared auth user's UID is stored in a junction table operator_email_group that maps the UID to each Operator profile ID. On sign-in, after auth succeeds, the frontend queries the junction to get available profiles. See DB Schema §08.

§ 09 — Database Schema

Schema Additions & Migrations

Three new tables and two modifications to existing tables are required. All migrations should be written as idempotent SQL with IF NOT EXISTS guards.

New Table: operator_email_group

Maps a shared Supabase auth UID to each Operator profile in that shared-email group.

CREATE TABLE IF NOT EXISTS operator_email_group (
  id          UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  auth_uid    UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  profile_id  UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ DEFAULT now(),
  UNIQUE (auth_uid, profile_id)
);

CREATE INDEX IF NOT EXISTS idx_oeg_auth_uid ON operator_email_group(auth_uid);

New Table: signup_audit

CREATE TABLE IF NOT EXISTS signup_audit (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  email           TEXT NOT NULL,
  designation     TEXT NOT NULL,
  operator_count  INT DEFAULT 1,
  plant_ids       UUID[] DEFAULT '{}',
  created_by      UUID REFERENCES user_profiles(id),
  created_at      TIMESTAMPTZ DEFAULT now()
);

New Table: plant_assignment_log

CREATE TABLE IF NOT EXISTS plant_assignment_log (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  target_user_id  UUID NOT NULL REFERENCES user_profiles(id),
  admin_id        UUID NOT NULL REFERENCES user_profiles(id),
  prev_plant_ids  UUID[] NOT NULL,
  new_plant_ids   UUID[] NOT NULL,
  justification   TEXT,
  created_at      TIMESTAMPTZ DEFAULT now()
);

Modified: login_attempts

ALTER TABLE login_attempts
  ADD COLUMN IF NOT EXISTS username   TEXT,
  ADD COLUMN IF NOT EXISTS plant_id   UUID REFERENCES plants(id),
  ADD COLUMN IF NOT EXISTS device_id  TEXT;

Modified: user_profiles — Operator single-plant trigger

CREATE OR REPLACE FUNCTION enforce_operator_single_plant()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.designation = 'Operator'
     AND array_length(NEW.plant_assignments, 1) > 1 THEN
    RAISE EXCEPTION 'Operators may only be assigned to a single plant';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_operator_single_plant
BEFORE INSERT OR UPDATE ON user_profiles
FOR EACH ROW EXECUTE FUNCTION enforce_operator_single_plant();
✓
Migration safety: Add this trigger and the operator_email_group table to the existing migrations_status.py tracking mechanism so Admins can confirm migration state from the MigrationsPanel in the Admin Console.
